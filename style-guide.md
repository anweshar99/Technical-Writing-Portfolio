# Documentation Style Guide (Sample)

## Purpose
This guide ensures consistency and clarity across technical documentation for the Mobile Data Collection system.

---

## Terminology

- Use "farmer record" (not "entry" or "data item")  
- Use "submit" for user actions (not "upload" or "send")  
- Use "field consultant" (not "user") in user-facing documentation  

---

## Capitalization

- Feature names should be capitalized when referring to UI elements  
  Example: "Navigate to the Records section"  

- Do not capitalize generic terms  
  Example: "submit the form"  

---

## Formatting

- Use inline code formatting for endpoints and parameters  
  Example: `/collect-data`, `farmer_id`  

- Use code blocks for JSON and API responses  

---

## Dates and Numbers

- Use ISO format: YYYY-MM-DD  
- Avoid ambiguous formats like DD/MM/YYYY  

---

## Error Messages

- Keep messages concise and actionable  
- Avoid vague phring such as "something went wrong"  
- Prefer: "Invalid farmer ID"  

---

## Clarity Guidelines

- Avoid combining multiple actions in one sentence  
- Separate system behavior from user instructions  
- Replace vague qualifiers ("properly", "as needed") with specific meaning  

---

## Versioning

- Use versioned endpoints when applicable  
  Example: `/v1/collect-data`  

---

## Notes
This guide reflects a structured approach to maintaining consistency in technical documentation.
